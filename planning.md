# Provenance Guard вҖ” Planning

Provenance Guard is a backend service a creative-sharing platform can plug into to
classify whether submitted text was written by a human or generated by AI, score how
confident that classification is, surface a plain-language transparency label, and let
creators appeal a decision they think is wrong. This document is written **before** the
implementation and is the primary reference used to prompt AI tools during the build.

---

## The five core questions

### 1. Detection signals

The pipeline uses **two genuinely distinct signals** вҖ” one semantic, one structural.

**Signal A вҖ” LLM classification (Groq, `llama-3.3-70b-versatile`).**
- *What it measures:* Whether the text *reads* as human or AI-generated, judged
  holistically вҖ” semantic coherence, register, the kind of "too-even" polish that
  large models produce.
- *Why it differs between human and AI:* AI text tends to be globally coherent,
  hedged, evenly structured, and free of the idiosyncrasies (typos, abrupt topic
  jumps, personal voice) that humans produce.
- *Output:* a JSON object `{ "ai_probability": 0.0вҖ“1.0, "reasoning": "..." }`.
- *Blind spot:* It can be fooled by lightly-edited AI text, by formal human writing
  that happens to read "machine-like" (academic prose), and by very short inputs where
  there isn't enough signal. It is also non-deterministic and costs an API call.

**Signal B вҖ” Stylometric heuristics (pure Python, no external libraries).**
- *What it measures:* Measurable statistical structure of the text:
  - **Sentence-length variance (burstiness):** humans vary sentence length a lot;
    AI text is more uniform.
  - **Type-token ratio (lexical diversity):** vocabulary richness over total words.
  - **Punctuation density:** punctuation marks per word.
  - **Mean sentence length.**
- *Why it differs:* AI text trends toward low burstiness and middling, consistent
  diversity; human text is more variable in both directions.
- *Output:* a normalized `ai_probability` in 0.0вҖ“1.0 plus the raw metrics, so the
  audit log and README can show exactly what was measured.
- *Blind spot:* It is fooled by formal/edited human writing (low burstiness looks
  "AI") and by short or highly-repetitive human text. It has no understanding of
  *meaning* at all вҖ” a coherent essay and word-salad with the same statistics score
  the same.

These are distinct because one is **semantic** (does it *mean* like a human wrote it)
and one is **structural** (is it *shaped* like human writing). They fail in different
places, so combining them is more informative than either alone.

### 2. Uncertainty representation

The system produces a single **AI-probability `p` in [0,1]**: the combined estimate
that the text is AI-generated. We deliberately design what the number *means* before
deciding how to compute it.

- `p вүҲ 0.5` means **genuine uncertainty** вҖ” the two signals disagree, or both are
  weakly committed. It is not "50% of the text is AI"; it is "we cannot tell."
- Combining: weighted average of the two signals. When the LLM signal is available it
  carries weight **0.6** and stylometry **0.4** (the LLM is the stronger holistic
  judge). When the LLM signal is **unavailable** (no API key / API error), we fall back
  to stylometry alone but **shrink the result toward 0.5** (`p вҶ’ 0.5 + 0.6В·(p_styвҲ’0.5)`)
  so a single weak signal can never produce a high-confidence verdict.
- **False-positive asymmetry:** on a writing platform, labeling a *human's* work as AI
  is worse than missing AI text. So the bar to declare "AI" is set **higher** than the
  bar to declare "human," and the uncertain band is wide. We never assert "AI" off a
  single structural signal.

Thresholds that separate the three label variants:

| Combined `p`        | Attribution      | Label variant         |
|---------------------|------------------|-----------------------|
| `p вүҘ 0.70`          | `likely_ai`      | high-confidence AI    |
| `0.30 < p < 0.70`   | `uncertain`      | uncertain             |
| `p вүӨ 0.30`          | `likely_human`   | high-confidence human |

The **displayed confidence %** is the model's confidence *in the verdict it gives*:
`likely_ai` вҶ’ `p`; `likely_human` вҶ’ `1 вҲ’ p`; `uncertain` вҶ’ reported as a range, not a
single reassuring number.

### 3. Transparency label design

Three variants, written out verbatim. `{pct}` is the displayed confidence as a whole
percentage; `{score}` is the raw combined `p`.

- **High-confidence AI** (`p вүҘ 0.70`):
  > рҹӨ– **Likely AI-generated** вҖ” Our analysis suggests this text was probably produced
  > by an AI system (about {pct}% confidence). This is an automated estimate, not a
  > certainty. If you wrote this yourself, you can appeal this label.

- **High-confidence human** (`p вүӨ 0.30`):
  > вңҚпёҸ **Likely human-written** вҖ” Our analysis found no strong signs of AI generation
  > (about {pct}% confidence this is human-written). This is an automated estimate, not
  > a guarantee of authorship.

- **Uncertain** (`0.30 < p < 0.70`):
  > вқ“ **Uncertain origin** вҖ” Our signals disagree about whether this text is
  > human-written or AI-generated, so we're not labeling it either way (AI-likelihood
  > вүҲ {score}). Treat the origin as unverified.

The label leads with a plain-language verdict and an icon, states confidence in words a
non-technical reader understands, and вҖ” critically for the AI case вҖ” always names the
appeal path.

### 4. Appeals workflow

- **Who can appeal:** the creator of a submission, identified by the `content_id`
  returned at submission time.
- **What they provide:** `content_id` and free-text `creator_reasoning`.
- **What the system does:** looks up the original decision, sets the content's status
  to `under_review`, writes an **appeal** audit-log entry that sits alongside the
  original classification entry (same `content_id`), and returns a confirmation. No
  automated re-classification is performed вҖ” a human reviewer would pick it up.
- **What a reviewer sees:** by filtering the audit log on `content_id` they get the
  original attribution, confidence, both signal scores, and the creator's reasoning in
  one place.

### 5. Anticipated edge cases (handled poorly вҖ” named honestly)

1. **A short, repetitive poem with simple vocabulary** (e.g. a nursery-style verse):
   low type-token ratio and low sentence-length variance make the stylometric signal
   read it as AI, even though a human wrote it. Our shrink-toward-0.5 fallback and wide
   uncertain band reduce the damage, but this is a likely false positive.
2. **Formal, polished human writing** (academic abstract, legal prose): globally even
   and low-burstiness, so *both* signals can lean AI. This is the most dangerous case
   because the signals agree for the wrong reason.
3. **Lightly-edited AI text:** a human pass over AI output breaks the structural
   uniformity and softens the LLM's read, pushing genuinely-AI text into the uncertain
   band (false negative). We accept this вҖ” on a writing platform a false negative is
   less harmful than a false positive.

**Submission flow:** `POST /submit` вҶ’ Signal A (raw text вҶ’ `p_llm`) вҶ’ Signal B
(raw text вҶ’ `p_sty`) вҶ’ confidence scoring (both scores вҶ’ combined `p` + attribution) вҶ’
label builder (`p`, attribution вҶ’ label text) вҶ’ audit log (write classification entry)
вҶ’ response `{content_id, attribution, confidence, label, signals}`.

**Appeal flow:** `POST /appeal` (`content_id`, `creator_reasoning`) вҶ’ status update to
`under_review` вҶ’ audit log (write appeal entry alongside the original) вҶ’ response
`{content_id, status, message}`.

## AI Tool Plan

**M3 вҖ” submission endpoint + first signal.**
- *Provide:* the Detection-signals section (Signal A) + the architecture diagram + the
  `POST /submit` contract.
- *Ask for:* a Flask app skeleton with the `/submit` route stub and the Groq LLM signal
  function returning `{ai_probability, reasoning}`.
- *Verify:* call the signal function directly on a few inputs and confirm it returns a
  score in [0,1]; confirm `/submit` returns `content_id` before any real logic.

**M4 вҖ” second signal + confidence scoring.**
- *Provide:* the Detection-signals section (Signal B) + Uncertainty-representation
  section + diagram.
- *Ask for:* the stylometry function and the combine/threshold scoring logic.
- *Verify:* clearly-AI vs clearly-human inputs produce meaningfully different `p`; the
  thresholds in code match the table above; print both signal scores when intuition and
  output disagree.

**M5 вҖ” production layer.**
- *Provide:* the Transparency-label variants + Appeals-workflow section + diagram.
- *Ask for:* the label-generation function, the `POST /appeal` endpoint, Flask-Limiter
  config, and the complete audit-log schema.
- *Verify:* all three label variants are reachable; an appeal flips status to
  `under_review` and is logged against the original `content_id`; rate limiting returns
  `429` past the limit; `GET /log` shows вүҘ3 structured entries.

## Rate-limit reasoning

Limit on `POST /submit`: **10 per minute, 100 per day** per client IP.

A genuine creator submits their own work; even an active writer rarely posts more than a
handful of pieces an hour, so 10/min is far above honest use while a script trying to
flood the classifier (or burn the Groq quota) hits the wall immediately. The 100/day cap
is a second backstop against a slow-drip abuser that stays under the per-minute limit.
These are starting values for a single-tenant demo; a real deployment would key on
authenticated account rather than IP.
